**Source:** edge+auditor | **Location:** `tests/e2e/fixtures/side-panel.ts:112-116`

The implementation uses `file://${fixturePath}`, while some dev notes in the project suggest `page.route` might be preferred for content script injection consistency. The tests appear to pass, but the spec/notes should be clarified to match the committed code.

## Patch

**These are fixable code issues:**

### 3. Missing `startRecordingWithPage()` Method [CRITICAL]
**Source:** blind+edge+auditor | **Location:** `tests/e2e/delete-recording.spec.ts:10`

The test calls `panel.startRecordingWithPage()`, but this method is missing from the `SidePanelPage` helper class. This will cause the test to fail.
**Fix:** Add the helper method to `SidePanelPage` to encapsulate starting and opening the test page.

### 4. Architectural Violation: State Management in Component
**Source:** blind+edge | **Location:** `src/components/sop-app.ts:195-200`

`handleDeleteFromEditor` directly manipulates `this.ctrl.loadedRecording = null` and orchestrates navigation/listing. This violates encapsulation; the controller should manage its own state transitions.
**Fix:** Move the cleanup logic into a dedicated controller method (e.g., `this.ctrl.deleteAndNavigateHome()`).

### 5. Unawaited Async Controller Methods
**Source:** blind | **Location:** `src/components/sop-app.ts:195-200`

If `deleteRecording` or `listRecordings` return Promises, they are currently called without `await`, potentially leading to race conditions where the UI navigates before data is persisted/cleared.
**Fix:** Ensure async controller calls are properly awaited.

### 6. Missing View Transition on Delete
**Source:** edge | **Location:** `src/components/sop-app.ts:195-200`

`handleDeleteFromEditor` performs a raw navigation to `home`, while `handleBack` uses `document.startViewTransition`. This creates inconsistent UX for the same edit-to-home transition.
**Fix:** Wrap the navigation in the View Transition API pattern used elsewhere.

### 7. Brittle Test Delays (`waitForTimeout`)
**Source:** blind+edge | **Location:** `tests/e2e/critical-path.spec.ts`

Multiple tests use hardcoded `waitForTimeout` calls. These are slow and non-deterministic.
**Fix:** Replace with event-driven waits like `panel.waitForStepCount()` or waiting for specific DOM visibility.

### 8. Selector Deviation from Spec
**Source:** auditor | **Location:** `tests/e2e/`

The spec mandates verification of specific classes like `.contrast` and `.sop-btn-danger`, but some tests use more generic `getByRole` locators.
**Fix:** Update assertions to verify mandated styling classes to meet all Acceptance Criteria.

### 9. Missing Title Verification in Saved Recordings
**Source:** auditor | **Location:** `tests/e2e/saved-recordings.spec.ts`

Acceptance Criterion #3 requires verifying the recording title in the home view, but the current test only checks for visibility and step count.
**Fix:** Add an assertion to verify the recording title matches the one from the recording.

### 10. Missing Deletion Confirmation
**Source:** blind+edge | **Location:** `src/components/sop-app.ts`

The deletion flow is destructive and immediate with no "Are you sure?" confirmation or "Undo" path. While consistent with existing home-view deletion, it remains a UX risk.

### 11. Multi-Panel Sync Limitations
**Source:** edge | **Location:** `src/entrypoints/background.ts`

The background script tracks a single `activePanelPort`, meaning actions in one side panel might not immediately update others in a multi-window scenario.
