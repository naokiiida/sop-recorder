## Bad Spec

**These findings suggest the spec/dev notes should be amended:**

### 1. `openTestPage()` uses `file://` URL but dev notes claim `page.route` HTTP URL fix
**Source:** edge+auditor | **Location:** `tests/e2e/fixtures/side-panel.ts:112-116`

The dev notes state: *"Content script can't inject on `file://` URLs — fixed by serving test page via `page.route` at HTTP URL."* However, the actual implementation uses `file://${fixturePath}`. If the content script truly cannot inject on `file://` URLs, this means clicks on the test page will not be captured as SOP steps, and `waitForStepCount()` will time out. Yet the story claims all 8 tests pass. One of two things is true: (a) the dev notes are inaccurate and `file://` works, or (b) the tests pass due to something else (e.g. the content script can inject on `file://` with the right permissions). The spec/notes contradict the code.

**Suggested spec amendment:** Clarify whether `file://` or `page.route` is the actual approach, and update the dev notes to match the committed code.

### 2. `exportAsZip()` uses `waitForEvent('download')` but spec warns chrome.downloads is not interceptable
**Source:** auditor | **Location:** `tests/e2e/fixtures/side-panel.ts:76-80`, `critical-path.spec.ts:50-51`

Task 4 spec says: *"Click 'Export as ZIP' and verify no error appears (chrome.downloads uses native save dialog, not interceptable by Playwright)."* Yet the actual test intercepts the download via `page.waitForEvent('download')`. If this works, the spec note is wrong. If it doesn't work, the test will time out. Since the story claims 8/8 tests pass, the spec note appears inaccurate.

**Suggested spec amendment:** Remove the note about chrome.downloads not being interceptable, since the implementation proves it is.

## Patch

**These are fixable code issues:**

### 3. Missing `startRecordingWithPage()` method in `SidePanelPage` [CRITICAL]
**Source:** blind+edge+auditor | **Location:** `tests/e2e/fixtures/side-panel.ts` (missing), `tests/e2e/delete-recording.spec.ts:10` (calls it)

`delete-recording.spec.ts` calls `panel.startRecordingWithPage()` but this method does not exist in the `SidePanelPage` class. The class defines `startRecording()` and `openTestPage()` separately. The task spec explicitly lists `startRecordingWithPage()` as a required method. This is a compilation/runtime error — the delete-recording test is broken.

**Fix:** Add the method to `SidePanelPage`:
```ts
async startRecordingWithPage(): Promise<Page> {
  await this.startRecording();
  return this.openTestPage();
}
```

### 4. `editStepTitle()` uses `.first()` globally instead of scoping to the clicked card
**Source:** blind | **Location:** `tests/e2e/fixtures/side-panel.ts:64-70`

The method clicks `nth(index)` editable title correctly, but then grabs the input with `.first()` across all step cards. If editing step index > 0, it will fill the wrong input.

**Fix:** Scope the input locator to the same step card:
```ts
const card = this.page.locator('sop-step-card').nth(index);
const input = card.locator('input[type="text"]');
```

### 5. `handleDeleteFromEditor` does not clear `this.ctrl.steps`
**Source:** edge | **Location:** `src/components/sop-app.ts:195-200`

Compare with `handleBack()` which effectively starts fresh when navigating home. `handleDeleteFromEditor` nulls `loadedRecording` but leaves `steps` populated with data from the deleted recording. While `navigateTo('home')` switches the view, stale steps remain in memory.

**Fix:** Add `this.ctrl.steps = [];` before navigating.

### 6. `handleDeleteFromEditor` does not use View Transition API (unlike `handleBack`)
**Source:** edge | **Location:** `src/components/sop-app.ts:195-200` vs `247-258`

`handleBack()` wraps its navigation in `document.startViewTransition()` when available, but `handleDeleteFromEditor` does a raw transition. Inconsistent UX for the same edit-to-home navigation.

**Fix:** Wrap the navigation in the same View Transition pattern as `handleBack()`.

### 7. Hardcoded `waitForTimeout` delays instead of event-driven waits
**Source:** blind+edge | **Location:** `critical-path.spec.ts:17,20,22`, `saved-recordings.spec.ts:16,18`, `delete-recording.spec.ts:12`

Multiple tests use `waitForTimeout(1000)`, `waitForTimeout(1500)`, and `waitForTimeout(3000)` for content script injection and step capture. These are both slow (wasting time) and flaky (insufficient on slow CI). The page object already has `waitForStepCount()` which should be used between clicks.

### 8. `SidePanelPage.deleteRecording()` has no wait-after-click
**Source:** edge | **Location:** `tests/e2e/fixtures/side-panel.ts:89-91`

Unlike `stopRecording()` which waits for the view transition, `deleteRecording()` clicks and returns immediately. The calling test must handle its own waits, which is inconsistent with the rest of the page object API.

## Defer

**Pre-existing issues surfaced by this review (not caused by current changes):**

### 9. No confirmation dialog before destructive recording deletion
**Source:** blind+edge | **Location:** `src/components/sop-editor.ts:420-429`

The `handleDeleteRecording` in `sop-editor.ts` dispatches the event with no `confirm()` or custom dialog. A single click permanently deletes the recording. This is a pre-existing UX gap (the delete button existed before this change; the new handler just wires it up). Step deletion has an undo toast, but recording deletion does not.

### 10. `extensionId` extraction assumes URL format
**Source:** edge | **Location:** `tests/e2e/fixtures/extension.ts:28`

`background.url().split('/')[2]` assumes `chrome-extension://<id>/...`. More robust: `new URL(background.url()).hostname`. Pre-existing in the fixture, not introduced by this change.
