## Patch

These are fixable code issues:

### P1. Orphan element fallback test has weak assertions

**Source:** blind + edge + auditor
**File:** `tests/unit/core/selector-generator.test.ts` (orphan fallback test)
**Severity:** Medium

The test only asserts `typeof result.css === 'string'` and `typeof result.xpath === 'string'`. An empty string or garbage output passes. Verified against source: when `el` is not found in `parent.children`, `nthOfTypeIndex` at `selector-generator.ts:116` returns `0`, producing `span:nth-of-type(0)` — an invalid CSS pseudo-class (nth-of-type is 1-based). Similarly `xpathPositionIndex` at `:153` returns `0`, producing `/span[0]` — invalid XPath.

The test should assert the actual fallback values to document this behavior. Per story Dev Notes ("do not fix bugs, document them"), the `0` return is a production code issue (see D1), but the test should at minimum assert meaningful output or explicitly document the known-invalid output.

### P2. xpathEscape concat test doesn't verify output correctness

**Source:** blind + edge
**File:** `tests/unit/core/selector-generator.test.ts` (xpathEscape concat test)
**Severity:** Medium

For the mixed-quotes case (`it's a "test"`), the test only checks `toContain('concat(')` and `toContain("'")`. A broken implementation (wrong segment order, unclosed parens, trailing empty parts) would still pass. Should assert the full expected output string or at minimum validate the expression is structurally complete.

### P3. Missing-blob test doesn't verify count of "Screenshot unavailable" occurrences

**Source:** blind
**File:** `tests/unit/core/zip-exporter.test.ts` (missing blobs test)
**Severity:** Low

Two steps are created with missing blobs — one with empty `screenshotBlobKey` (skipped at `zip-exporter.ts:32`) and one where `fetchBlob` returns null (skipped at `:35`). Both should produce `*(Screenshot unavailable)*` in the markdown via `export-engine.ts:66`. But `toContain()` passes if the string appears only once. The test should assert the string appears **twice** to confirm both code paths result in the unavailable marker.

### P4. 50-step zero-padding test doesn't verify steps 2-9

**Source:** blind
**File:** `tests/unit/core/export-engine.test.ts` (50-step test)
**Severity:** Low

Checks `step-01`, `step-10`, `step-50` and negatively asserts `step-5.jpg` is absent. But never checks `step-02` through `step-09`. If padding only worked for step 1 (hardcoded special case) but was broken for steps 2-9, this test would still pass. The negative assertion should cover at least one more single-digit index (e.g., `step-3.jpg`).

### P5. fetchBlob mock doesn't distinguish empty-key vs null-fetch paths

**Source:** blind
**File:** `tests/unit/core/zip-exporter.test.ts` (missing blobs test)
**Severity:** Low

The first step has `screenshotBlobKey: ''`, causing `zip-exporter.ts:32` to `continue` before calling `fetchBlob`. The second step has key `'key-missing'` but `fetchBlob` returns null. Since the mock returns null unconditionally, the test can't verify whether the empty-key step correctly skips the fetch call. Should assert `fetchBlob` was called exactly once (for `'key-missing'` only).

## Defer

Pre-existing issues surfaced by this review (not caused by current changes):

### D1. `nthOfTypeIndex` / `xpathPositionIndex` fallback returns 0 — invalid selectors

**Source:** edge
**File:** `src/core/selector-generator.ts:116, :153`

When an element is not found in parent's children, `index` stays at `0`, producing `:nth-of-type(0)` (invalid CSS — 1-based) and `/span[0]` (invalid XPath — 1-based). Should return `1`. This is a production code bug but out of scope for this test-only story.

**Recommendation:** File as a separate bug story.

### D2. `labelForTag` mappings for `textarea`, `img`, `label` tags entirely untested

**Source:** edge
**File:** `src/core/step-manager.ts:46-48`

The `roleMap` includes `textarea` -> `'text area'`, `img` -> `'image'`, `label` -> `'label'`, but no test (existing or new) exercises these tag names in a click/dblclick context. A regression in the map would go unnoticed. Pre-existing gap.

### D3. `labelForTag` with `input` + `elementType` in click context and `select` -> `dropdown` untested

**Source:** edge
**File:** `src/core/step-manager.ts:44-45`

Clicking an `<input type="text">` should produce `"Clicked 'X' text input"` via the dynamic `elementType` interpolation, and clicking a `<select>` should produce `"Clicked 'X' dropdown"`. Neither combination is tested — the `check` test uses `elementType` but via the `check` generator, not `click`, and `select` tests use the `select` action, not `click`. Pre-existing gap.

### D4. Duplicate IDs edge case in step-manager untested

**Source:** auditor (AC #2)
**File:** `src/core/step-manager.ts`

AC #2 explicitly lists "duplicate IDs" as an edge case to test. No test in the existing or new suite calls `loadSteps` with steps sharing the same `id`, or otherwise exercises duplicate-ID behavior. Whether the `StepManager` should reject, deduplicate, or accept duplicates is unspecified. Pre-existing gap.
